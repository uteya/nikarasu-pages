Section 1 — WHY

なぜClaude Codeを使うのか

問題: 「優秀なAIがいるのに、仕事が進まない」

Gemini や ChatGPT は強力なAIだが、「成果物への変換が手動」という根本的な限界がある。

  • ⚠会話が消える — 毎回「前回の続きだけど...」から始まる。文脈を繰り返し説明
  • ⚠ファイルを触れない — テキスト生成だけ。実際のファイル作成・編集は手動コピペ
  • ⚠Gitと連携しない — コミット・ブランチ・PRは全部自分で操作
  • ⚠プロジェクト構造を知らない — 毎回「このプロジェクトは...」と説明が必要
  • ⚠チームで共有できない — 会話は個人のもの。他メンバーが参照できない

結果: AIとの会話に時間をかけたのに、成果物への変換が手動。

解決: Claude Code = 永続記憶 + ファイル操作 + Git + 外部ツール連携

CLAUDE.md + STATUS.md でプロジェクト文脈を永続保持。毎回説明する必要がない。

にからすが選んだ5つの理由

  • 3事業・3リポジトリの横断管理 — 各PJの状態がSTATUS.mdに書かれ、CCが起動時に自動で読む
  • チームコラボレーション — CLAUDE.md（共有ルール）+ CLAUDE.local.md（個人設定）の階層で同じ品質
  • 非同期ワークフロー — Issue → @claude → PR → マージ。時間も場所も問わない
  • 「仕様→実装」の一気通貫 — ドキュメント→コード→テスト→コミットが1セッションで完結
  • MCP連携 — Googleカレンダー、Slack、freee等と直接連携可能

ペりん向け: 併用ガイド

Geminiを既に使いこなしているペりんにとって、Claude Codeは「対立するツール」ではなく「補完するツール」。

  • Geminiが得意なこと — 自由なブレインストーミング、広範な知識ベースからの情報検索、マルチモーダル入力
  • Claude Codeが得意なこと — プロジェクトコンテキストを維持した上での実装、ファイル操作、Git連携、チーム成果物の管理
  • 実践的な使い分け — Geminiでアイデアを固める → Claude Codeで実装・ドキュメント化・コミット。「考える」と「作る」を分離する
Section 3 — HOW

アーキテクチャ

4層 CLAUDE.md 階層設計

CCを起動すると、これら4層が自動で読み込まれる。毎回説明する必要がない。

Layer 4: Global chezmoi管理
~/.claude/CLAUDE.md

グローバル設定 — セッション開始/終了ルーチン、日本語応答、チーム規約。全PJ共通で適用。

Layer 1: Project Git管理 (共有)
CLAUDE.md (PJルート)

チーム共有ルール — チーム構成、ブランチ戦略、コミット規約、参照ドキュメント。60行以下。

Layer 2: Personal .gitignore対象
CLAUDE.local.md

個人設定 — 自分の役割、担当領域、作業スタイル。他メンバーには共有されない。

Layer 3: Rules (将来用) Git管理
.claude/rules/*.md

パス限定ルール — モノレポ時にディレクトリ別のルールを定義。現在は未使用。

.context/ — プロジェクトの記憶

STATUS.md
プロジェクトの「現在地」。フェーズ、進捗率、メンバー別タスク、次のマイルストーン。CCが毎回起動時に読む。

PM管理小竹が更新。他メンバーは読み取り専用。

CHANGES.md
変更履歴。いつ、誰が、何をしたかが時系列で記録される。作業完了時に追記する。

全員追記Append-Only。既存エントリは消さない。

SPECS.md
仕様・要件。プロダクトの仕様書やアーキテクチャ設計を記録。

PM管理変更時はPRで通知。

Section 4 — Quick Start

Section 6 — Rules

5つのルール

  1. 1
    STATUS.md はPM（小竹）が管理 — 編集しない プロジェクトの状態はPMが責任を持つ。参照は自由。提案はPRで。
  2. 2
    CHANGES.md は追記のみ — 既存エントリを消さない 履歴は積み重ね。誰の記録も消さないこと。Append-Only。
  3. 3
    PRにはレビュアーを指定する 作成したPRには相手をレビュアーに設定。内容確認を経てからマージ。
  4. 4
    main に直接pushしない（小竹以外）— PR経由で mainは保護ブランチ。全ての変更はPRを経由する。
  5. 5
    困ったらIssueで相談 判断に迷うこと、ブロッカー、疑問は全てIssueに書く。記録が残り、後から参照できる。
Section 7 — @claude

@claude の使い方

claude-code-action とは

@claude は GitHub の Issue コメントや PR コメントに書くことで Claude Code を呼び出す仕組み。リポジトリに設定された GitHub Actions ワークフロー（claude-code-action）が自動で起動し、ブランチ作成・ファイル編集・PR作成まで行う。

Issue の本文に書いても、コメント欄に書いてもOK。PRのレビューコメントでも使える。

依頼の書き方のコツ

  • 具体的に書く — 「資料を作って」より「Q1の売上データを表形式でまとめて」
  • ファイル名を指定する — 「docs/report.md に書いて」と出力先を明示
  • 1つの依頼に1つのタスク — 複数タスクは別Issueに分ける方が精度が上がる
  • 修正はPRコメントで — PRのコメント欄に @claude で修正を依頼できる

実行例

例 1: レポート作成
@claude docs/report.md にQ1の売上データを表形式でまとめて。前年比も入れてください。
指定ファイルにデータを構造化して書き込む
例 2: 仕様書の修正
@claude この仕様書の「対象ユーザー」セクションを中小企業向けに修正して。具体的なペルソナを2つ追加してください。
既存ファイルの特定セクションを編集する
例 3: ステータス更新
@claude .context/CHANGES.md に「2026-03-12: GTM戦略書v4.0レビュー完了」を追記して。
変更履歴への追記（CHANGES.md は全員が追記可能）
例 4: 調査依頼
@claude docs/research/ に中小企業DX市場の調査レポートを作成して。市場規模、主要プレイヤー、トレンドをカバーしてください。
新規ファイルを作成して調査結果をまとめる

にからすのCC基盤は6つのバージョンを経て進化する。

v1 3月 今ここ

WBS + ダッシュボード + 手動CC運用 — STATUS.md + CHANGES.md + CLAUDE.md の基本形。CCが毎回読み、文脈を引き継ぐ。WBS（wbs-complete.md）でタスク管理し、HTMLダッシュボードで可視化。

v2 4月 次のステップ

Agent Teams + 並列タスク実行 — Issue → @claude（GitHub Actions） → 自動でブランチ+実装+PR。人間はレビューしてマージするだけ。複数タスクが並列で進む。

v3 5-6月

会議 → 議事録 → WBS自動更新パイプライン — Zoom会議 → 録音 → 書き起こし → 議事録生成 → 決定事項をdecisions.mdに追記 → WBSのタスク・期限を自動更新 → 担当者に通知。

v4 Q3

ポータル階層化（経営→事業→PJ→個人） — 上のアーキテクチャ図が実現。経営ポータルから各事業・PJ・個人まで、ドリルダウンで見られる。

v5 Q4

秘書CC x N人（各メンバー専属AI） — 各メンバーに専属のClaude Codeが付き、個人のスタイル・文脈・優先度を完全に把握。秘書CC同士がコミュニケーションし、チーム全体が有機的に動く。

v6 Year 2 ビジョン

AI経営OSとしてプロダクト化・外販 — にからす自身の運用実績をベースに、この仕組みを他社に提供する「AI経営OS」プロダクトへ。中小企業が3人チームでも大企業並みの経営管理を実現する世界。

Section 9 — Onboarding

メンバー追加手順

新しいメンバーがチームに参加する際の手順。PM（小竹）が実施する。

1
GitHub Org 招待（nikarasu-inc）

GitHub Orgの Settings → People → Invite member で新メンバーを招待。必要なリポジトリへのWrite権限を付与する。

2
Claude Code 契約確認

Max Plan推奨。APIキーの発行が必要な場合は設定方法を案内する。

3
環境セットアップ（パスを選択）

A) CLIゼロ開始 → GitHub Webだけで開始（上記Quick Start参照）
B) フル開始 → CLI インストール + リポジトリclone + CLAUDE.local.md作成

4
CLAUDE.local.md 作成

テンプレート（CLAUDE.local.md.template）をコピーして、名前・役割・担当領域・作業スタイルを記入。.gitignore 対象なのでリポジトリにはpushされない。

5
WBS + ダッシュボードに反映

docs/wbs-complete.md に担当タスクを追加。docs/dashboard.html に名前を追加。CLAUDE.md のチーム表も更新。

6
初回体験: Issue → @claude → PR → マージ

テスト用のIssueを作成し、@claude で依頼。PR が作成されたらレビューしてマージ。このフローを一通り体験する。

チェックリスト

  • GitHub Org 招待 → 承認済み
  • Claude Code 契約確認 → APIキー or Max Plan
  • CC インストール → claude --version 成功（CLI利用の場合）
  • CLAUDE.local.md 作成 → 個人設定記入
  • WBS にタスク追加 → コミット
  • ダッシュボードに反映 → GitHub Pages 確認
  • CLAUDE.md チーム表更新 → コミット
  • 初回 Issue → @claude → PR → マージ 成功
Section 10 — FAQ

よくある質問

以下を確認してください:
1. リポジトリに .github/workflows/claude-code.yml が存在するか
2. GitHub リポジトリの Settings → Secrets に ANTHROPIC_API_KEY が設定されているか
3. Actions タブでワークフローが有効になっているか
4. コメントに @claude が正確に含まれているか（大文字小文字に注意）

解決しない場合は Issue で小竹に相談してください。
PR のコメント欄に @claude [修正内容] と書けば、Claude が修正を加えてコミットします。具体的に書くほど精度が上がります。

例: @claude 表の3列目を「前年比」から「成長率(%)」に変更して
GitHub Web 上でファイルを開き、右上の鉛筆アイコン（Edit this file）をクリックすれば直接編集できます。編集後は「Commit changes」で保存。

ただし main ブランチに直接コミットせず、「Create a new branch for this commit and start a pull request」を選択して PR 経由でマージしてください。
複数ファイルにまたがる大きな変更や、ビルド・テストが必要な作業は Desktop App / CLI の導入を検討してください。

Windows: winget install Anthropic.ClaudeCode
Mac: brew install claude-code または npm install -g @anthropic-ai/claude-code

インストール後: リポジトリ clone → CLAUDE.local.md 作成 → claude 起動。
もちろん。アイデア出し・ブレスト・リサーチにはGeminiを使い、実装・ファイル作成・Git操作・チーム成果物の管理にはClaude Codeを使う。「考える」と「作る」を分離するのが効率的。

ポイント: チームで共有する成果物（コード、ドキュメント、設計書）は必ずClaude Code経由でGitにコミットすること。Geminiで考えたことも、最終的にはCCで実体化する流れ。
MCP（Model Context Protocol）はClaude Codeを外部サービスと接続する仕組み。これにより、CCから直接Googleカレンダーの予定を確認したり、Slackにメッセージを送ったりできる。

現在稼働中:
• Google Calendar — 会議予定の確認・作成

将来導入予定:
• Slack — メッセージ送受信・通知
• freee — 会計データ参照
• Gmail — メール送受信

設定は .mcp.json（PJ別、Git管理）または ~/.claude/settings.json（グローバル）に記載する。
